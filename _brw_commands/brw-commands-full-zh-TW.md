# BRW 指令 — 完整規格

所有 BRW 指令的完整 prompt 規格。這是人類閱讀用的參考文件 — 實際的指令 prompt 在 `commands/` 資料夾中。

---

## 共用：`_brw-init`

所有指令開始時都先執行這段初始化邏輯。

### Vault 偵測

1. 檢查當前工作目錄是否有 `.obsidian/` — 如果有，這裡就是 vault
2. 如果沒有，讀取 `~/.claude/CLAUDE.md` 中的 `## Vault` 段落的 `path` 設定
3. 如果都沒有，問使用者 vault 路徑，確認包含 `.obsidian/`，然後存入 `~/.claude/CLAUDE.md`

### 語言偵測

檢查 vault 的 `CLAUDE.md` 中 `## Vault Structure` 下的 `language` 設定。

如果沒有，問使用者，選項：
- English（預設）
- 繁體中文
- 日本語
- 其他（自行輸入）

存入 vault 的 `CLAUDE.md`。之後所有輸出使用這個語言。

### 資料夾偵測

每個用途按順序匹配第一個存在的候選資料夾：

| 用途 | 候選名稱 |
|---|---|
| daily-notes | `Daily/`, `_daily/`, `daily/`, `Journal/`, `journal/` |
| inbox | `Inbox/`, `_inbox/`, `inbox/`, `_Capture/`, `Capture/` |
| analysis | `_brw_analysis/`, `_claude_code_analysis/`, `Analysis/`, `_analysis/`, `analysis/` |
| logs | `_brw_logs/`, `_claude_code_feedback/`, `Logs/`, `_logs/`, `logs/` |
| templates | `Templates/`, `_templates/`, `templates/` |

找不到時：問使用者、建立資料夾、印出完整路徑、存入 vault 的 `CLAUDE.md` 的 `## Vault Structure`。

之後再執行時，直接讀取 `## Vault Structure` 的快取設定，不再偵測。

### 內容資料夾

「內容資料夾」= vault 根目錄下的所有第一層資料夾，排除：
- 以 `.` 開頭的資料夾（`.obsidian/`、`.claude/`、`.git/` 等）
- `## Vault Structure` 中列出的系統資料夾

如果沒有任何內容資料夾，改為掃描 vault 根目錄的 `.md` 檔案。

## 日常節奏

### `/brw-status`

**用途：** Vault 現況摘要。
**輸入：** 無。
**需要的資料夾：** daily-notes + 所有內容資料夾。

掃描：
- 所有內容資料夾（深度 2 層）
- daily-notes 最近 7 天

輸出：

**內容主題** — 每個內容資料夾及子目錄：名稱、筆記數、草稿數、最近修改日期。狀態：活躍（7 天內有修改）/ 休眠。

**近期活動** — 最近 7 天修改的檔案，按時間倒序。每筆：檔名、所屬資料夾、修改時間、變更性質（新建 / 修改 / 大幅改寫）。上限 15 筆。

**Vault 概覽** — 總檔案數、內容資料夾數、最近 7 天活躍的資料夾。

空的區塊顯示「無」而非省略。

---

### `/brw-catchup [天數]`

**用途：** 進度銜接。
**輸入：** 可選天數（預設 1）。
**需要的資料夾：** daily-notes、inbox。

讀取：
- 最近的 daily note（預設昨天，指定天數時讀取多天）
- 所有 inbox 檔案

邏輯：找到每個 daily note 中最後一個 Wrap 區塊，提取「進行中」和「下一步」。

輸出：

**近期進度** — 最後一個 Wrap 的進行中和下一步。如果沒有 daily note 或沒有 Wrap 區塊，標註「無進度紀錄」。多天時：列出每天最後一個 Wrap，按時間倒序。

**Inbox 待處理** — Inbox 檔案：檔名、建立日期、一句話摘要。上限 10 筆。空的話：「Inbox 清空」。

**建議焦點** — 1-3 個今天值得推進的事項。每個附理由（為什麼是今天做）。最多 3 個。

---

### `/brw-wrap [描述]`

**用途：** 進度快照寫入 daily note。
**輸入：** 可選重點描述。
**需要的資料夾：** daily-notes、templates。

掃描：
- 本次 session 的檔案操作（建立、修改、刪除）
- 今天的 daily note

寫入目標：`{daily-notes}/YYYY-MM-DD.md`
- 檔案不存在且 `templates/daily.md` 存在：用模板建立
- 檔案不存在且無模板：建立只含 `# YYYY-MM-DD` 標題的檔案
- 檔案已存在：在末尾追加新 Wrap 區塊，用水平線分隔

每次 Wrap 的格式：

```
---

## Wrap [HH:MM]

### 完成
- [這個時段完成的具體事項，一筆一句話]

### 進行中
- [已開始但未完成的事項。如果沒有，標註「無」]

### 下一步
- [接下來要做的事。如果暫時沒有，標註「待定」]
```

規則：
- 時間戳使用本地時間 HH:MM（24 小時制）
- 不修改之前的 Wrap 區塊，只追加
- 如果有描述輸入，用它來組織 Wrap 內容
- 寫入前顯示預覽 + 完整路徑，確認後再執行
- 寫入後印出實際寫入的路徑

### `/brw-weekreview`

**用途：** 週回顧與落差分析。
**輸入：** 無。
**需要的資料夾：** daily-notes + 所有內容資料夾。

讀取：
- 所有內容資料夾最近 7 天的修改紀錄
- 最近 7 天的 daily notes

輸出：

**本週進度** — 每個有活動的內容資料夾：名稱、推進了什麼（根據檔案變更）、下一步。

**近期動態** — 上週新增或修改的筆記，哪些接近完成（如果 drafts/ 有內容）。

**落差分析** — 比對 daily notes 中的優先事項 vs 實際檔案變更：
- 說了重要但沒動手（聲稱優先但無對應檔案變更）
- 沒提到但花了時間（有檔案變更但 daily notes 未提及）

如果 daily notes 不足 3 天，標註「紀錄不足，落差分析可能不準確」。

## 概念考古

### `/brw-arc {概念}`

**用途：** 追蹤一個概念在 vault 中的演化軌跡。
**輸入：** 必填 — 一個概念或關鍵詞。
**需要的資料夾：** analysis。

掃描：整個 vault，優先搜尋 daily-notes 和內容資料夾。

邏輯：
1. 檢查 `{analysis}/arc/` 中今天是否有匹配的檔案
2. 有 → 追加模式
3. 疑似相關 → 列出候選，問使用者
4. 沒有 → 建立模式

**建立模式** — 寫入 `{analysis}/arc/YYYY-MM-DD-{概念}.md`：

```
# 概念追蹤：{概念}

> 分析時間：YYYY-MM-DD HH:MM

## 首次出現
最早提到這個概念的檔案和脈絡。引用不超過 3 句。

## 演化時間軸
每個提到這個概念的檔案，按時間排列：
- YYYY-MM-DD HH:MM | [[檔名]] | 一句話摘要

## 目前狀態
這個概念連結到哪些主題、有無矛盾用法、有無正在發展的 draft。

## 空白地帶
這個概念有哪些面向在 vault 中尚未被探討。
```

**追加模式** — 在既有檔案末尾追加：

```
---

## 追蹤更新 [HH:MM]

### 演化時間軸（新增）
- [自上次分析以來新出現的提及]

### 狀態變化
- [有什麼變化。如果沒有，標註「無顯著變化」]

### 空白地帶（更新）
- [重新評估尚未探討的面向]
```

規則：
- 超過 20 個檔案時，列出最重要的 20 個並標註總數
- 所有檔案引用使用 [[wiki link]] 格式
- 時間戳使用本地時間 HH:MM（24 小時制）
- 寫入前顯示預覽，確認後執行，寫入後印出路徑

### `/brw-bridge {主題A} 與 {主題B}`

**用途：** 找出兩個主題之間的隱藏連結。
**輸入：** 必填 — 兩個主題（分隔詞：「與」或對應語言的等價詞）。
**需要的資料夾：** analysis。

掃描：整個 vault。

邏輯：
1. 檢查 `{analysis}/bridge/` 中今天是否有匹配的檔案（不分順序）
2. 有 → 追加模式
3. 疑似 → 列出候選，問使用者
4. 沒有 → 建立模式

**建立模式** — 寫入 `{analysis}/bridge/YYYY-MM-DD-{主題A}-與-{主題B}.md`：

```
# 橋接分析：{主題A} 與 {主題B}

> 分析時間：YYYY-MM-DD HH:MM

## 直接交集
同時提到兩個主題的檔案。每筆：[[檔名]]、兩個主題在該檔案中如何被關聯。
如果沒有，標註「無直接交集」。

## 橋樑筆記
只提到其中一個主題、但可能構成連結的檔案。說明為什麼它可能是橋樑。
上限 5 筆。每筆用 [[wiki link]]。

## 潛在連結假設
1-3 個深層連結假設。每個標註信心程度（強 / 中 / 弱）和依據。
```

**追加模式：**

```
---

## 追蹤更新 [HH:MM]

### 新發現的交集
- [自上次分析以來新出現的交集或橋樑筆記]

### 假設驗證
- [之前的假設是否有新的支持或反駁證據]
```

### `/brw-undercurrent [範圍]`

**用途：** 發掘潛流主題。
**輸入：** 可選範圍（特定目錄）。
**需要的資料夾：** analysis、daily-notes、inbox + 所有內容資料夾。

掃描（無範圍時）：所有內容資料夾、所有 daily-notes、所有 inbox。

邏輯：
1. 檢查 `{analysis}/undercurrent/` 中今天是否有檔案
2. 有 → 追加模式
3. 沒有 → 建立模式

**建立模式** — 寫入 `{analysis}/undercurrent/YYYY-MM-DD.md`：

```
# 潛流分析

> 分析時間：YYYY-MM-DD HH:MM
> 掃描範圍：[列出實際掃描的目錄]

## 潛流主題

### 1. {主題/詞彙}
- 出現次數：{N} 次，涉及 {M} 個檔案
- 出現脈絡：
  - [[檔名A]]：[一句話脈絡]
  - [[檔名B]]：[同上]
  （上限 5 個檔案）
- 判斷：[值得展開嗎？為什麼？]

### 2. {主題/詞彙}
...
```

判斷標準：
- 出現在 3 個以上不同檔案中
- 沒有以它為主題的獨立文章或 draft
- 不是常見功能詞或結構詞

**追加模式：**

```
---

## 追蹤更新 [HH:MM]

### 新發現的潛流
- [之前未列出的主題，格式同上]

### 已有潛流的變化
- [之前列出的主題，出現頻率或脈絡有無變化]
```

規則：按出現頻率倒序排列。上限 10 個主題。

**用途：** 想法群聚分析。
**輸入：** 可選範圍。
**需要的資料夾：** analysis、daily-notes、inbox + 所有內容資料夾。

掃描（無範圍時）：所有內容資料夾、daily-notes（最近 30 天）、所有 inbox。

邏輯：同 undercurrent（檢查 `{analysis}/cluster/` 今天是否有檔案）。

**建立模式** — 寫入 `{analysis}/cluster/YYYY-MM-DD.md`：

```
# 想法群聚分析

> 分析時間：YYYY-MM-DD HH:MM
> 掃描範圍：[列出實際掃描的目錄]

## 想法群

### 1. {建議的群組名稱}
- 涉及筆記：
  - [[檔名A]]（修改日期：YYYY-MM-DD）
  - [[檔名B]]（修改日期：YYYY-MM-DD）
- 共同主題：[2-3 句摘要]
- 成熟度：高 / 中 / 低
- 建議發展方向：文章 / 專案 / 概念框架 / 繼續累積

### 2. {建議的群組名稱}
...
```

判斷標準：
- 3 篇以上筆記涉及相似概念但沒有共同的上層分類
- 撰寫時間分布在不同日期（不是一次性產出）

**追加模式：**

```
---

## 追蹤更新 [HH:MM]

### 新發現的想法群
- [新的想法群，格式同上]

### 已有想法群的變化
- [有無新筆記加入、成熟度是否提升]
```

規則：上限 5 個想法群，按成熟度倒序。

## 思考工具

### `/brw-ghost {問題} [save]`

**用途：** 用你的語氣和立場回答問題。
**輸入：** 必填問題。末尾加 `save` 寫入檔案。
**需要的資料夾：** analysis + 所有內容資料夾。

掃描：所有內容資料夾，排除 `drafts/` 和 `thoughts/` 子目錄（只看完成的文章）。

步驟：
1. 分析寫作風格：句式偏好、用詞習慣、論述結構、常用修辭
2. 提取與問題相關的既有立場和論點
3. 用這個風格和立場組成回答

輸出：

**回答** — 用你的語氣。長度與你的文章段落長度一致。

**依據** — 引用的筆記：[[檔名]]、引用的具體段落或立場。上限 5 筆。

**信心標註** — 哪些部分有明確的 vault 依據，哪些是風格推測的延伸。

寫入邏輯（僅當輸入含 `save` 時）：
1. 移除 "save"，剩餘部分作為問題
2. 檢查 `{analysis}/ghost/` 中今天是否有匹配的檔案
3. 有 → 追加 `## 追加回答 [HH:MM]` 區塊
4. 沒有 → 建立 `{analysis}/ghost/YYYY-MM-DD-{問題摘要}.md`

### `/brw-stress-test {主題} [save]`

**用途：** 對一個主題的論述進行壓力測試。
**輸入：** 必填主題。末尾加 `save` 寫入檔案。
**需要的資料夾：** analysis。

掃描：vault 中所有提到該主題的檔案。

步驟：
1. 收集所有關於該主題的論點和主張
2. 逐一檢驗每個主張的邏輯和證據

輸出：

**主張清單** — 找到的所有主要主張。每筆：一句話主張、來源 [[檔名]]。

**弱點分析** — 對每個主張，檢查（適用的才列）：
- 內部矛盾：不同檔案對同一件事說法不一致
- 未驗證假設：主張建立在未被證明的前提上
- 邏輯跳躍：論證中缺少中間步驟
- 證據缺口：主張缺乏支撐的資料或案例
每個弱點引用具體的 [[檔名]] 和段落。

**整體評估** — 穩固程度：穩固 / 大致穩固但有缺口 / 需要重大補強。列出最需要優先處理的 1-3 個弱點。

寫入邏輯（僅當輸入含 `save` 時）：
1. 移除 "save"，剩餘部分作為主題
2. 檢查 `{analysis}/stress-test/` 中今天是否有匹配的檔案
3. 有 → 追加 `## 追蹤更新 [HH:MM]` 區塊
4. 沒有 → 建立 `{analysis}/stress-test/YYYY-MM-DD-{主題摘要}.md`

### `/brw-harvest [save]`

**用途：** 從既有內容中收割可行動的想法。
**輸入：** 可選 `save` 寫入檔案。
**需要的資料夾：** analysis、daily-notes、inbox + 所有內容資料夾。

掃描：所有內容資料夾、daily-notes（最近 30 天）、所有 inbox。

輸出 — 四個類別，每類上限 5 個：

**可以寫的文章** — 有足夠素材可以發展成文章的主題。每筆：建議標題、素材來源（[[檔名]] 列表）、缺什麼才能動筆。

**可以做的工具或專案** — 工具需求、流程痛點、明確的產品想法。每筆：描述、來源、預估複雜度（小 / 中 / 大）。

**值得研究的主題** — 被提及但未深入探討的外部概念、理論或技術。每筆：主題、vault 中提及的脈絡、為什麼值得研究。

**可以聯繫的人或社群** — 提到的人物、組織或社群，且與目前工作方向相關。每筆：名稱、vault 中提及的脈絡、聯繫的理由。如果沒有，標註「無」。

寫入邏輯（僅當輸入含 `save` 時）：
1. 檢查 `{analysis}/harvest/` 中今天是否有檔案
2. 有 → 追加 `## 追蹤更新 [HH:MM]` 區塊
3. 沒有 → 建立 `{analysis}/harvest/YYYY-MM-DD.md`

### `/brw-graduate`

**用途：** 讓半成形想法畢業為獨立筆記。
**輸入：** 無。
**需要的資料夾：** daily-notes、logs、inbox + 所有內容資料夾。

掃描：
- daily-notes（最近 14 天）— 從 Wrap 區塊中找反覆出現在「下一步」或「進行中」的卡住項目
- logs（最近 14 天）— 從 Journal 中找靈感連結、改進方向、未完成項目
- inbox（全部）
- 所有內容資料夾，遞迴搜尋名稱匹配 `draft*` 或 `thought*` 的子目錄（不分大小寫）

畢業標準：
- 有明確的核心主張（不只是隨想或問句）
- 與 vault 中至少一個既有主題相關
- 有足夠的內容密度（不只是一句話）

**第一階段：候選清單** — 列出符合條件的想法：
- 來源檔案與位置（哪個段落）
- 核心主張摘要（一句話）
- 建議歸屬的內容資料夾
- 與哪些既有筆記相關

列出後暫停，等使用者確認哪些要畢業。

**第二階段：執行畢業**（確認後才執行）— 對每個確認的想法，在對應的內容資料夾建立新的 markdown 檔案：
- 標題
- 核心主張
- 出處脈絡（從哪個 daily note 或 inbox 項目來的）
- 與 vault 其他筆記的關聯（[[wiki link]]）
- 待展開的方向

寫入前顯示完整路徑，確認後執行，寫入後印出路徑。

## 維護

### `/brw-tidy [fix]`

**用途：** 屬性健檢與精簡。
**輸入：** 可選 — 輸入 `fix` 確認後自動修正。
**需要的資料夾：** 所有內容資料夾 + inbox + daily-notes。

掃描：
- 所有內容資料夾（遞迴）
- inbox — 全部檔案
- daily-notes — 最近 30 天

**檢查項目：**

1. **缺漏** — 沒有 frontmatter 的檔案、缺少 `tags` 或 `status`
2. **一致性** — 無效的 `status`/`source` 值（source 允許：self, claude-code, agent, external）、tag 大小寫不一致（`AI-alignment` vs `ai-alignment`）、底線 vs 連字號不一致
3. **冗餘** — 只出現 1 次的 tag（可能拼錯）、超過 6 個 tags 的檔案、`topic` 與 tag 重複、`created` 格式不一致
4. **Tag 收斂** — 使用頻率統計、語意相近的合併候選、前 10 核心標籤、孤立標籤

**輸出：** 報告 + 健康度分數（無問題檔案數 / 總檔案數）。

```
# 屬性健檢報告

> 檢查時間：YYYY-MM-DD HH:MM
> 掃描範圍：[實際掃描的目錄]
> 掃描檔案數：N

## 缺漏問題（N 個）
- [[檔名]] — 缺少 frontmatter / 缺少 tags / 缺少 status

## 一致性問題（N 個）
- [[檔名]] — status 值 "xxx" 不在允許範圍
- Tag 大小寫不一致：`AI-alignment`（3 次）vs `ai-alignment`（5 次）→ 建議統一為 `ai-alignment`

## 冗餘問題（N 個）
- [[檔名]] — 有 8 個 tags，建議精簡
- Tag `xxx` 只出現 1 次，於 [[檔名]]

## Tag 總覽
### 核心標籤（前 10）
| Tag | 使用次數 |
|-----|---------|

### 建議合併
- `ai_alignment` → `ai-alignment`（共 N 個檔案需更新）

### 孤立標籤（僅出現 1 次）
| Tag | 出現於 |
|-----|--------|

## 摘要
- 需修正：N 個檔案
- 建議精簡：N 個標籤
- 健康度：N%
```

**修正模式**（輸入包含 `fix` 時）：

可自動修正（顯示完整變更清單、確認後執行）：
- 補上缺少的 frontmatter 骨架（空 tags + `status: draft`）
- 統一 tag 大小寫（以多數為準）
- 統一底線為連字號
- 移除與 tag 重複的 `topic`

需逐一確認：
- 合併語意相近的 tags
- 精簡超過 6 個 tags 的檔案
- 刪除孤立標籤

修正後重新計算健康度，輸出差異摘要。

**寫入行為：** 總是將報告寫入 `{analysis}/tidy/YYYY-MM-DD.md`。同天追加更新，含變更摘要和健康度差異。

每次最多報告 50 個問題。超過的標註總數，建議分批處理。

## 紀錄

### `/brw-journal {描述}`

**用途：** 工作紀錄與思考文章。
**輸入：** 必填 — 主題描述。
**需要的資料夾：** logs + 所有內容資料夾（用於搜尋相關筆記）。

掃描：
- 本次 session 的操作紀錄
- vault 中與主題相關的筆記
- logs 資料夾中今天的既有 journal 檔案

邏輯：
1. 掃描 `{logs}/` 中 `YYYY-MM-DD-*.md` 檔案（今天日期）
2. 比對檔名和標題與輸入主題的關鍵詞
3. 明確匹配 → 追加模式
4. 疑似相關 → 列出候選，問使用者
5. 沒有匹配 → 建立模式

**建立模式** — 寫入 `{logs}/YYYY-MM-DD-{主題摘要}.md`：

```
# {主題}

> 建立時間：YYYY-MM-DD HH:MM

## 背景
這件事的起因。如果 vault 中有前情筆記，用 [[wiki link]] 引用。

## 做了什麼
具體操作和決策紀錄，有因果敘述的段落。

## 為什麼這樣做
關鍵決策的理由。有取捨的話，說明選了什麼、放棄了什麼。

## 靈感連結
與其他想法的關聯、延伸可能、跨主題的連結。[[wiki link]] 指向相關筆記。如果沒有，標註「無」。

## 可能的改進方向
尚未驗證但值得探索的方向、替代方案、潛在風險。如果沒有，標註「無」。

## 未完成與後續
需跟進事項。都完成了就標註「無待辦」。
```

**追加模式** — 在末尾追加：

```
---

## 追加紀錄 [HH:MM]

### 做了什麼
[新增的操作與決策]

### 為什麼這樣做
[新增決策的理由]

### 靈感連結
[新的關聯。如果沒有，標註「無」]

### 可能的改進方向
[新的方向。如果沒有，標註「無」]
```

同時檢查「未完成與後續」區塊 — 如果有項目已完成，標註完成時間。
