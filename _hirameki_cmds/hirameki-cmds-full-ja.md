# Hirameki コマンド — 完全仕様

すべての Hirameki コマンドの完全な prompt 仕様。人間が読むためのリファレンスドキュメント — 実際のコマンド prompt は `commands/` フォルダ内にあります。

---

## 共通：`__init`

すべてのコマンドは `~/.claude/CLAUDE.md` の `## Vault Structure` セクションを読み込んで開始します。

### Vault 検出

1. 現在の作業ディレクトリに `.obsidian/` があるか確認 — あればここが vault
2. なければ `~/.claude/CLAUDE.md` の `## Vault` セクションの `path` 設定を読む
3. どちらもなければユーザーに vault のパスを尋ね、`.obsidian/` が含まれることを確認後、`~/.claude/CLAUDE.md` に保存

### 言語設定

`~/.claude/CLAUDE.md` の `## Vault Structure` 内の `language` 設定を確認。

なければユーザーに質問。選択肢：
- 繁體中文
- English
- 日本語
- その他（自由入力）

`~/.claude/CLAUDE.md` に保存。以降すべての出力はこの言語を使用。

### フォルダ検出

各用途について、最初に見つかった候補フォルダを使用：

| 用途 | 候補名 |
|---|---|
| daily-notes | `Daily/`, `_daily/`, `daily/`, `Journal/`, `journal/` |
| inbox | `Inbox/`, `_inbox/`, `inbox/`, `_Capture/`, `Capture/` |
| analysis | `_hirameki_analysis/`, `_agent_analysis/`, `_claude_code_analysis/`, `Analysis/`, `_analysis/`, `analysis/` |
| logs | `_hirameki_logs/`, `_agent_logs/`, `_claude_code_feedback/`, `Logs/`, `_logs/`, `logs/` |
| templates | `Templates/`, `_templates/`, `templates/` |

見つからない場合：ユーザーに確認・フォルダ作成・完全パスを表示・`~/.claude/CLAUDE.md` の `## Vault Structure` に保存。

次回以降は `## Vault Structure` のキャッシュ設定を直接読み込み、再検出しない。

### コンテンツフォルダ

「コンテンツフォルダ」= vault ルート直下の第 1 層フォルダのうち、以下を除いたすべて：
- `.` で始まるフォルダ（`.obsidian/`、`.claude/`、`.git/` など）
- `## Vault Structure` にリストされているシステムフォルダ

コンテンツフォルダがない場合は vault ルートの `.md` ファイルをスキャン。

## 日常リズム

### `/hirameki:status`

**用途：** Vault の現況サマリー。
**入力：** なし。
**必要なフォルダ：** daily-notes + すべてのコンテンツフォルダ。

スキャン：
- すべてのコンテンツフォルダ（深さ 2 層）
- daily-notes 直近 7 日

出力：

**コンテンツテーマ** — 各コンテンツフォルダとサブディレクトリ：名前・ノート数・ドラフト数・最終更新日。状態：アクティブ（7 日以内に更新）/ 休眠。

**最近の活動** — 直近 7 日に更新されたファイル、更新時間の降順。各エントリ：ファイル名・所属フォルダ・更新時間・変更の性質（新規 / 修正 / 大幅改訂）。上限 15 件。

**Vault 概要** — 総ファイル数・コンテンツフォルダ数・直近 7 日にアクティブなフォルダ数。

空のセクションは省略せず「なし」と表示。

---

### `/hirameki:catchup [日数]`

**用途：** 進捗の引き継ぎ。
**入力：** オプションで日数（デフォルト 1）。
**必要なフォルダ：** daily-notes、inbox。

読み込み：
- 直近の daily note（デフォルトは昨日、日数指定時は複数日）
- すべての inbox ファイル

ロジック：各 daily note の最後の Wrap ブロックを見つけ、「進行中」と「次のステップ」を抽出。

出力：

**最近の進捗** — 最後の Wrap の進行中と次のステップ。daily note または Wrap ブロックがない場合は「進捗記録なし」と表示。複数日の場合：各日の最後の Wrap を時間降順で列挙。

**Inbox 未処理** — Inbox ファイル：ファイル名・作成日・一言要約。上限 10 件。空なら「Inbox クリア」。

**フォーカス提案** — 今日推進すべき 1〜3 件。各件に理由付き（なぜ今日やるべきか）。最大 3 件。

---

### `/hirameki:wrap [説明]`

**用途：** 進捗スナップショットを daily note に書き込む。
**入力：** オプションで重点説明。
**必要なフォルダ：** daily-notes、templates。

スキャン：
- 今回のセッションのファイル操作（作成・修正・削除）
- 当日の daily note

書き込み先：`{daily-notes}/YYYY-MM-DD.md`
- ファイルがなく `templates/daily.md` がある：テンプレートから作成
- ファイルがなくテンプレートもない：`# YYYY-MM-DD` 見出しだけのファイルを作成
- ファイルが存在：末尾に新しい Wrap ブロックを水平線で区切って追記

各 Wrap のフォーマット：

```
---

## Wrap [HH:MM]

### 完了
- [この時間帯に完了した具体的な事項、1 件 1 文]

### 進行中
- [開始したが未完了の事項。なければ「なし」]

### 次のステップ
- [次にやること。当面なければ「未定」]
```

ルール：
- タイムスタンプはローカル時間 HH:MM（24 時間制）
- 以前の Wrap ブロックは変更せず、末尾に追記のみ
- 説明入力がある場合、それを基に Wrap の内容を整理
- 書き込み前にプレビューと完全パスを表示し、確認後に実行
- 書き込み後に実際の書き込みパスを表示

---

### `/hirameki:weekreview`

**用途：** 週次レビューとギャップ分析。
**入力：** なし。
**必要なフォルダ：** daily-notes + すべてのコンテンツフォルダ。

読み込み：
- すべてのコンテンツフォルダの直近 7 日の変更履歴
- 直近 7 日の daily notes

出力：

**今週の進捗** — 活動があった各コンテンツフォルダ：名前・何を進めたか（ファイル変更に基づく）・次のステップ。

**最近の動向** — 先週追加・修正されたノートで、完成に近いもの（drafts/ にコンテンツがあれば）。

**ギャップ分析** — daily notes の優先事項 vs 実際のファイル変更を比較：
- 重要と言ったのに手をつけていない（宣言したが対応するファイル変更がない）
- 言及していないのに時間を使っている（ファイル変更があるが daily notes に記載なし）

daily notes が 3 日未満の場合は「記録不足、ギャップ分析が不正確な可能性あり」と表示。

## 概念考古学

### `/hirameki:arc {概念}`

**用途：** Vault 内での概念の進化を追跡。
**入力：** 必須 — 概念またはキーワード。
**必要なフォルダ：** analysis。

スキャン：Vault 全体、daily-notes とコンテンツフォルダを優先。

ロジック：
1. `{analysis}/arc/` 内に今日付けの一致するファイルがあるか確認
2. あれば → 追記モード
3. 関連性が疑われる → 候補を列挙し、ユーザーに確認
4. なければ → 新規作成モード

**新規作成モード** — `{analysis}/arc/YYYY-MM-DD-{概念}.md` に書き込み：

```
# 概念トラッキング：{概念}

> 分析時間：YYYY-MM-DD HH:MM

## 初出
この概念が最初に登場したファイルと文脈。3 文以内で引用。

## 進化タイムライン
この概念に言及している各ファイル、時系列順：
- YYYY-MM-DD HH:MM | [[ファイル名]] | 一言要約

## 現在の状態
この概念がどのテーマと結びついているか、矛盾した使い方がないか、進行中のドラフトがあるか。

## 空白地帯
この概念の中で Vault でまだ探求されていない側面。
```

**追記モード** — 既存ファイルの末尾に追記：

```
---

## トラッキング更新 [HH:MM]

### 進化タイムライン（追加）
- [前回の分析以降に新たに登場した言及]

### 状態の変化
- [何か変化があれば。なければ「顕著な変化なし」]

### 空白地帯（更新）
- [未探求の側面の再評価]
```

ルール：
- 20 ファイルを超える場合は最重要 20 件を列挙し、総数を記載
- すべてのファイル引用は [[wiki link]] 形式
- タイムスタンプはローカル時間 HH:MM（24 時間制）
- 書き込み前にプレビューを表示し、確認後に実行、書き込み後にパスを表示

### `/hirameki:bridge {テーマA} と {テーマB}`

**用途：** 2 つのテーマ間の隠れたつながりを発見。
**入力：** 必須 — 2 つのテーマ（区切り語：「と」または対応言語の等価語）。
**必要なフォルダ：** analysis。

スキャン：Vault 全体。

ロジック：
1. `{analysis}/bridge/` 内に今日付けの一致するファイルがあるか確認（順序不問）
2. あれば → 追記モード
3. 疑われる → 候補を列挙し、ユーザーに確認
4. なければ → 新規作成モード

**新規作成モード** — `{analysis}/bridge/YYYY-MM-DD-{テーマA}-と-{テーマB}.md` に書き込み：

```
# ブリッジ分析：{テーマA} と {テーマB}

> 分析時間：YYYY-MM-DD HH:MM

## 直接の交点
両方のテーマに言及しているファイル。各エントリ：[[ファイル名]]・そのファイルで 2 つのテーマがどう関連しているか。
なければ「直接の交点なし」。

## 橋渡しノート
一方のテーマにしか言及していないが、つながりを構成しうるファイル。なぜ橋渡しになりうるかを説明。
上限 5 件。各エントリは [[wiki link]]。

## 潜在的なつながり仮説
1〜3 個の深層つながり仮説。各仮説に信頼度（強 / 中 / 弱）と根拠を記載。
```

**追記モード：**

```
---

## トラッキング更新 [HH:MM]

### 新たに発見された交点
- [前回の分析以降に新たな交点や橋渡しノートが見つかれば]

### 仮説の検証
- [以前の仮説を支持または反証する新たな証拠があれば]
```

### `/hirameki:undercurrent [スコープ]`

**用途：** 潜流テーマの発掘。
**入力：** オプションでスコープ（特定ディレクトリ）。
**必要なフォルダ：** analysis、daily-notes、inbox + すべてのコンテンツフォルダ。

スキャン（スコープなし時）：すべてのコンテンツフォルダ・すべての daily-notes・すべての inbox。

ロジック：
1. `{analysis}/undercurrent/` 内に今日付けのファイルがあるか確認
2. あれば → 追記モード
3. なければ → 新規作成モード

**新規作成モード** — `{analysis}/undercurrent/YYYY-MM-DD.md` に書き込み：

```
# 潜流分析

> 分析時間：YYYY-MM-DD HH:MM
> スキャン範囲：[実際にスキャンしたディレクトリを列挙]

## 潜流テーマ

### 1. {テーマ/キーワード}
- 出現回数：{N} 回、{M} ファイルに登場
- 出現文脈：
  - [[ファイル名A]]：[一言で文脈を説明]
  - [[ファイル名B]]：[同上]
  （上限 5 ファイル）
- 判断：[展開する価値があるか？なぜ？]

### 2. {テーマ/キーワード}
...
```

判断基準：
- 3 つ以上の異なるファイルに登場
- それを主題とした独立した記事やドラフトがない
- 一般的な機能語や構造語ではない

**追記モード：**

```
---

## トラッキング更新 [HH:MM]

### 新たに発見された潜流
- [以前リストになかったテーマ、同じフォーマットで]

### 既存の潜流の変化
- [以前リストしたテーマの出現頻度や文脈の変化]
```

ルール：出現頻度の降順で並べる。上限 10 テーマ。

```
# アイデアクラスター分析

> 分析時間：YYYY-MM-DD HH:MM
> スキャン範囲：[実際にスキャンしたディレクトリを列挙]

## アイデアグループ

### 1. {提案するグループ名}
- 関連ノート：
  - [[ファイル名A]]（更新日：YYYY-MM-DD）
  - [[ファイル名B]]（更新日：YYYY-MM-DD）
- 共通テーマ：[2〜3 文で要約]

...
```

- 3 本以上のノートが類似した概念を扱っているが共通の上位カテゴリがない
- 執筆時間が異なる日付に分布している（一時的な産出ではない）

**追記モード：**

```
---

## トラッキング更新 [HH:MM]

### 新たに発見されたクラスター
- [新しいアイデアグループ、同じフォーマットで]

### 既存クラスターの変化
- [新しいノートが加わったか、成熟度が上がったか]
```

ルール：上限 5 グループ、成熟度の降順で並べる。

## 思考ツール

### `/hirameki:ghost {質問} [save]`

**用途：** あなたの語り口とスタンスで質問に回答。
**入力：** 必須で質問。末尾に `save` でファイルに保存。
**必要なフォルダ：** analysis + すべてのコンテンツフォルダ。

スキャン：すべてのコンテンツフォルダ、`drafts/` と `thoughts/` サブディレクトリを除外（完成した記事のみ参照）。

手順：
1. 文章スタイルを分析：文体の好み・用語の習慣・論述構造・よく使うレトリック
2. 質問に関連する既存のスタンスと論点を抽出
3. そのスタイルとスタンスを用いて回答を構成

出力：

**回答** — あなたの語り口で。長さはあなたの記事の段落長に合わせる。

**根拠** — 引用したノート：[[ファイル名]]・引用した具体的な段落やスタンス。上限 5 件。

**信頼度の注記** — どの部分に明確な Vault の根拠があり、どの部分がスタイルによる推測の延長か。

書き込みロジック（入力に `save` が含まれる場合のみ）：
1. "save" を除いた残りが質問
2. `{analysis}/ghost/` 内に今日付けの一致するファイルがあるか確認
3. あれば → `## 追加回答 [HH:MM]` ブロックを追記
4. なければ → `{analysis}/ghost/YYYY-MM-DD-{質問の要約}.md` を作成

### `/hirameki:stress-test {テーマ} [save]`

**用途：** テーマの論述に圧力テストを行う。
**入力：** 必須でテーマ。末尾に `save` でファイルに保存。
**必要なフォルダ：** analysis。

スキャン：vault 内でそのテーマに言及しているすべてのファイル。

手順：
1. そのテーマに関するすべての論点と主張を収集
2. 各主張のロジックと証拠を順番に検証

出力：

**主張リスト** — 見つかったすべての主要な主張。各エントリ：一文で主張・ソース [[ファイル名]]。

**弱点分析** — 各主張について（該当するもののみ列挙）：
- 内部矛盾：異なるファイルで同じことについて言い方が食い違っている
- 未検証の前提：主張が証明されていない前提の上に成り立っている
- 論理の飛躍：論証に中間ステップが欠けている
- 証拠の欠如：主張を裏付けるデータや事例が不足している
各弱点には具体的な [[ファイル名]] と段落を引用。

**総合評価** — 堅牢度：堅牢 / おおむね堅牢だが欠缺あり / 大幅な補強が必要。優先的に対処すべき 1〜3 つの弱点を列挙。

書き込みロジック（入力に `save` が含まれる場合のみ）：
1. "save" を除いた残りがテーマ
2. `{analysis}/stress-test/` 内に今日付けの一致するファイルがあるか確認
3. あれば → `## トラッキング更新 [HH:MM]` ブロックを追記
4. なければ → `{analysis}/stress-test/YYYY-MM-DD-{テーマの要約}.md` を作成

### `/hirameki:harvest [save]`

**用途：** 既存コンテンツから実行可能なアイデアを収穫。
**入力：** オプションで `save` を入力してファイルに保存。
**必要なフォルダ：** analysis、daily-notes、inbox + すべてのコンテンツフォルダ。

スキャン：すべてのコンテンツフォルダ・daily-notes（直近 30 日）・すべての inbox。

出力 — 4 カテゴリ、各カテゴリ上限 5 件：

**書ける記事** — 十分な素材があって記事に発展できるテーマ。各エントリ：提案タイトル・素材ソース（[[ファイル名]] リスト）・執筆を始めるために何が不足しているか。

**作れるツールまたはプロジェクト** — ツールのニーズ・プロセスの痛点・明確なプロダクトアイデア。各エントリ：説明・ソース・予想複雑度（小 / 中 / 大）。

**研究すべきテーマ** — 言及されているが深く探求されていない外部の概念・理論・技術。各エントリ：テーマ・Vault での言及文脈・なぜ研究する価値があるか。

**連絡すべき人またはコミュニティ** — 言及されている人物・組織・コミュニティで現在の作業方向と関連するもの。各エントリ：名前・Vault での言及文脈・連絡の理由。なければ「なし」。

書き込みロジック（入力に `save` が含まれる場合のみ）：
1. `{analysis}/harvest/` 内に今日付けのファイルがあるか確認
2. あれば → `## トラッキング更新 [HH:MM]` ブロックを追記
3. なければ → `{analysis}/harvest/YYYY-MM-DD.md` を作成

### `/hirameki:graduate`

**用途：** 半成形のアイデアを独立ノートに卒業させる。
**入力：** なし。
**必要なフォルダ：** daily-notes、logs、inbox + すべてのコンテンツフォルダ。

スキャン：
- daily-notes（直近 14 日）— Wrap ブロックの「次のステップ」や「進行中」に繰り返し登場する行き詰まりアイテムを探す
- logs（直近 14 日）— Journal のインスピレーションのつながり・改善の方向性・未完了アイテムを探す
- inbox（すべて）
- すべてのコンテンツフォルダ、`draft*` または `thought*` に名前が一致するサブディレクトリを再帰的に検索（大小文字区別なし）

卒業基準：
- 明確な核心主張がある（随想や疑問文だけではない）
- Vault 内の少なくとも 1 つの既存テーマと関連している
- 十分なコンテンツ密度がある（1 文だけではない）

**フェーズ 1：候補リスト** — 条件を満たすアイデアを列挙：
- ソースファイルと場所（どの段落か）
- 核心主張の要約（一文）
- 推奨する配置先コンテンツフォルダ
- 関連する既存ノート

列挙後に一時停止し、ユーザーが卒業させるものを確認するのを待つ。

**フェーズ 2：卒業の実行**（確認後のみ実行）— 各確認済みアイデアについて、対応するコンテンツフォルダに新しい markdown ファイルを作成：
- タイトル
- 核心主張
- 出典の文脈（どの daily note や inbox アイテムから来たか）
- Vault の他のノートとの関連（[[wiki link]]）
- 展開すべき方向性

書き込み前に完全なパスを表示し、確認後に実行、書き込み後にパスを表示。

## メンテナンス

### `/hirameki:tidy [fix]`

**用途：** プロパティの健全性チェックと整理。
**入力：** オプション — `fix` を入力すると確認後に自動修正。
**必要なフォルダ：** すべてのコンテンツフォルダ + inbox + daily-notes。

スキャン：
- すべてのコンテンツフォルダ（再帰的）
- inbox — すべてのファイル
- daily-notes — 直近 30 日

**チェック項目：**

1. **欠落** — frontmatter がないファイル・`tags` または `status` が欠けているファイル
2. **一貫性** — 無効な `status`/`source` 値（source の許容値：self, claude-code, agent, external）・タグの大小文字の不整合（`AI-alignment` vs `ai-alignment`）・アンダースコアとハイフンの不整合
3. **冗長性** — 1 回しか登場しないタグ（タイポの可能性）・タグが 6 個超のファイル・`topic` とタグの重複・`created` フォーマットの不整合
4. **タグの収束** — 使用頻度の統計・意味的に近いタグの統合候補・上位 10 コアタグ・孤立タグ

**出力：** レポート + 健全性スコア（問題のないファイル数 / 総ファイル数）。

```
# プロパティ健全性レポート

> チェック時間：YYYY-MM-DD HH:MM
> スキャン範囲：[実際にスキャンしたディレクトリ]
> スキャンファイル数：N

## 欠落の問題（N 件）
- [[ファイル名]] — frontmatter なし / tags なし / status なし

## 一貫性の問題（N 件）
- [[ファイル名]] — status 値 "xxx" が許容範囲外
- タグの大小文字不整合：`AI-alignment`（3 回）vs `ai-alignment`（5 回）→ `ai-alignment` に統一推奨

## 冗長の問題（N 件）
- [[ファイル名]] — タグが 8 個、整理を推奨
- タグ `xxx` は 1 回だけ登場、[[ファイル名]] にて

## タグ概要
### コアタグ（上位 10）
| タグ | 使用回数 |
|-----|---------|

### 統合推奨
- `ai_alignment` → `ai-alignment`（N ファイルの更新が必要）

### 孤立タグ（1 回のみ登場）
| タグ | 登場ファイル |
|-----|--------|

## サマリー
- 修正が必要：N ファイル
- 整理を推奨：N タグ
- 健全性：N%
```

**修正モード**（入力に `fix` が含まれる場合）：

自動修正可能（完全な変更リストを表示し、確認後に実行）：
- 欠けている frontmatter の骨格を追加（空の tags + `status: draft`）
- タグの大小文字を統一（多数派に合わせる）
- アンダースコアをハイフンに統一
- タグと重複する `topic` を削除

個別確認が必要：
- 意味的に近いタグの統合
- タグが 6 個超のファイルの整理
- 孤立タグの削除

修正後に健全性スコアを再計算し、差分サマリーを出力。

**書き込み動作：** 常に `{analysis}/tidy/YYYY-MM-DD.md` にレポートを書き込む。同日は追記更新、変更サマリーと健全性の差分を含む。

1 回のレポートで最大 50 件の問題を報告。超えた場合は総数を記載し、分割処理を推奨。

## 記録

### `/hirameki:journal {説明}`

**用途：** 作業ログと思考記事。
**入力：** 必須 — テーマの説明。
**必要なフォルダ：** logs + すべてのコンテンツフォルダ（関連ノートの検索用）。

スキャン：
- 今回のセッションの操作ログ
- テーマに関連する Vault 内のノート
- logs フォルダ内の今日付けの既存 journal ファイル

ロジック：
1. `{logs}/` 内の `YYYY-MM-DD-*.md` ファイル（今日付け）をスキャン
2. ファイル名とタイトルを入力テーマのキーワードと照合
3. 明確に一致 → 追記モード
4. 関連性が疑われる → 候補を列挙し、ユーザーに確認
5. 一致なし → 新規作成モード

**新規作成モード** — `{logs}/YYYY-MM-DD-{テーマの要約}.md` に書き込み：

```
# {テーマ}

> 作成時間：YYYY-MM-DD HH:MM

## 背景
このことの発端。Vault に前史のノートがあれば [[wiki link]] で引用。

## 何をしたか
具体的な操作と意思決定の記録、因果関係のある段落で。

## なぜそうしたか
主要な意思決定の理由。トレードオフがあれば、何を選び何を諦めたかを説明。

## インスピレーションのつながり
他のアイデアとの関連・延長の可能性・クロステーマのつながり。[[wiki link]] で関連ノートを指す。なければ「なし」。

## 改善の可能性
未検証だが探求する価値のある方向性・代替案・潜在的なリスク。なければ「なし」。

## 未完了と後続
フォローアップが必要な事項。すべて完了したら「未了事項なし」。
```

**追記モード** — 末尾に追記：

```
---

## 追加記録 [HH:MM]

### 何をしたか
[追加した操作と意思決定]

### なぜそうしたか
[追加した意思決定の理由]

### インスピレーションのつながり
[新たな関連。なければ「なし」]

### 改善の可能性
[新たな方向性。なければ「なし」]
```

同時に「未完了と後続」セクションを確認 — 完了した項目があれば完了時間を記載。
