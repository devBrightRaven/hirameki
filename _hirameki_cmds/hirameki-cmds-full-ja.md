# Hirameki コマンド — 完全仕様

すべての Hirameki コマンドの完全な prompt 仕様。これは人間が読むためのリファレンス — 実際のコマンド prompt は `commands/` フォルダにあります。

---

## 共通：`__init`

すべてのコマンドは最初に `~/.claude/CLAUDE.md` の `## Vault Structure` セクションを読み込みます。

### Vault 検出

1. 現在の作業ディレクトリに `.obsidian/` があるか確認 — あればそこが vault
2. なければ `~/.claude/CLAUDE.md` の `## Vault` セクションの `path` 設定を読む
3. どちらもなければ、ユーザーに vault パスを尋ね、`.obsidian/` を含むことを確認して `~/.claude/CLAUDE.md` に保存

### 言語検出

`~/.claude/CLAUDE.md` の `## Vault Structure` 下の `language` 設定を読む。すべての出力はこの言語で行う。

### フォルダ検出

各用途について、候補から最初に存在するフォルダをマッチ：

| 用途 | 候補 |
|------|------|
| daily-notes | `Daily/`, `_daily/`, `daily/`, `Journal/`, `journal/` |
| inbox | `Inbox/`, `_inbox/`, `inbox/`, `_Capture/`, `Capture/` |
| analysis | `_hirameki_analysis/`, `_agent_analysis/`, `_claude_code_analysis/`, `Analysis/`, `_analysis/`, `analysis/` |
| logs | `_hirameki_logs/`, `_agent_logs/`, `_claude_code_logs/`, `Logs/`, `_logs/`, `logs/` |
| templates | `Templates/`, `_templates/`, `templates/` |

見つからない場合：ユーザーに確認、フォルダを作成、完全パスを表示、`~/.claude/CLAUDE.md` の `## Vault Structure` に保存。

2回目以降は `## Vault Structure` のキャッシュ設定を直接読み込む。

### コンテンツフォルダ

「コンテンツフォルダ」= vault ルート直下の全フォルダから以下を除いたもの：
- ドット始まりのフォルダ（`.obsidian/`・`.claude/`・`.git/` など）
- `_hirameki_cmds/`
- `## Vault Structure` に記載されているシステムフォルダ

コンテンツフォルダがない場合は vault ルートの全 `.md` ファイルをスキャン。

## セッション開始

### `/hirameki:catchup [日数]`

**目的：** 進捗の引き継ぎ。
**入力：** オプション日数（デフォルト 1）。
**必要なフォルダ：** daily-notes・inbox。

読み込み：
- 最近の daily notes（デフォルト：昨日、または指定日数分）
- すべての inbox ファイル

ロジック：各 daily note の最後の Wrap ブロックを見つける。「進行中」と「次のステップ」を抽出。

出力：

**最近の進捗** — 最後の Wrap の進行中と次のステップ。daily note や Wrap ブロックがない場合は「進捗記録なし」と注記。複数日の場合：各日の最後の Wrap を時系列逆順でリスト。

**Inbox 保留中** — inbox ファイル：ファイル名・作成日・1 文の要約。上限 10 件。空の場合：「Inbox クリア」。

**フォーカス提案** — 今日推進する価値がある 1〜3 件。各件に理由付き（なぜ今日なのか）。最大 3 件。

## セッション終了

### `/hirameki:wrap [説明]`

**目的：** 進捗スナップショットを daily note に書き込む。
**入力：** オプションの重点説明。
**必要なフォルダ：** daily-notes・templates。

スキャン：
- 現在のセッションのファイル操作（作成・変更・削除）
- 今日の daily note

書き込み先：`{daily-notes}/YYYY-MM-DD.md`
- ファイルなし + `templates/daily.md` あり → テンプレートを使用
- ファイルなし + テンプレートなし → `# YYYY-MM-DD` ヘッダーで作成
- ファイルあり → 水平線区切りで末尾に新しい Wrap ブロックを追記

各 Wrap のフォーマット：

```
---

## Wrap [HH:MM]

### 完了
- [この期間に完了した具体的な項目、各1文]

### 進行中
- [開始したが未完了の項目。なければ「なし」と記入]

### 次のステップ
- [次にやること。未決定なら「未定」と記入]
```

ルール：
- タイムスタンプはローカル時間 HH:MM（24 時間制）
- 前の Wrap ブロックは変更せず、追記のみ
- 入力に説明がある場合はそれを使って Wrap の内容を整理
- 書き込み前にプレビューと完全パスを表示、確認後に実行
- 実行後に実際の書き込みパスを表示

## Vault の状態確認

### `/hirameki:status [week|patterns]`

**目的：** Vault の現況（3 つのモード）。
**入力：** なし / `week` / `patterns`
**必要なフォルダ：** daily-notes + 全コンテンツフォルダ（patterns モードは inbox も必要）。

#### 引数なしモード — 即時スナップショット

スキャン：
- 全コンテンツフォルダ（深さ 2 層）
- daily-notes 直近 7 日

出力：

**コンテンツテーマ** — 各コンテンツフォルダとサブフォルダ：名前・ノート数・ドラフト数・最終更新日。ステータス：アクティブ（7 日以内に更新）/ 休眠。

**最近の活動** — 直近 7 日に更新されたファイル、時系列逆順。各件：ファイル名・親フォルダ・更新時刻・変更種類（新規作成 / 修正 / 大幅改稿）。上限 15 件。

**Vault 概要** — 総ファイル数・コンテンツフォルダ数・直近 7 日でアクティブなフォルダ。

空のセクションは「なし」と表示（省略しない）。

#### `status week` — 週次レビューとギャップ分析

読み込み：
- 全コンテンツフォルダの直近 7 日の更新記録
- 直近 7 日の daily notes

出力：

**今週の進捗** — アクティブな各コンテンツフォルダ：名前・何を推進したか（ファイル変更から判断）・次のステップ。

**最近の動き** — 先週追加・更新されたノート、完成に近いもの。

**ギャップ分析** — daily notes の優先事項と実際のファイル変更を比較：
- 重要と言ったのに着手しなかった（優先と宣言したが対応するファイル変更なし）
- 言及しなかったのに時間を使った（ファイル変更はあるが daily notes で未言及）

3 日分未満の daily notes なら「記録不足、ギャップ分析が不正確な可能性あり」と注記。

#### `status patterns` — 潜流とクラスター分析

スキャン：全コンテンツフォルダ・daily-notes（直近 30 日）・全 inbox。

**潜流テーマ** — 繰り返し登場するが独立した記事のないテーマ。基準：3 つ以上の異なるファイルに出現、独立した記事や draft なし。各テーマ：
- テーマ名
- 出現数と関連ファイル数（最大 5 件を [[wiki link]] で引用）
- 評価：展開する価値がある？なぜ？
上限 10 件、出現頻度の降順。

**形成中のクラスター** — 共通の親カテゴリなしに類似概念を扱う 3 本以上のノートが異なる日付で書かれているグループ。各グループ：
- 提案するクラスター名
- 関連ノート（[[wiki link]]、上限 5 件）
- 共通テーマの要約（2〜3 文）
- 成熟度：高 / 中 / 低
- 推奨の発展方向：記事 / プロジェクト / 概念フレームワーク / 蓄積継続
上限 5 件、成熟度の降順。

## 特定のアイデアを深掘り

### `/hirameki:explore {入力} [save]`

**目的：** 概念の発掘 — 入力の形からモードを自動判断。
**入力：** 必須。形がモードを決定。末尾に `save` を追加でファイルに書き込み。
**必要なフォルダ：** analysis + 全コンテンツフォルダ。

#### モード判定

| 入力の形 | モード |
|----------|--------|
| 単一の概念またはキーワード | **Arc** |
| 2 つのテーマを「と」「and」「與」で区切る | **Bridge** |
| `?` や `？` で終わる | **Ghost** |
| `test:` で始まる | **Stress-test** |

#### Arc モード — 概念進化の追跡

スキャン：vault 全体、daily-notes とコンテンツフォルダを優先。

ロジック：
1. `{analysis}/arc/` で今日の同概念ファイルを確認
2. マッチあり → 追記モード
3. 曖昧なマッチ → 候補をリスト、ユーザーに確認
4. マッチなし → 新規作成モード

**新規作成モード** — `{analysis}/arc/YYYY-MM-DD-{概念}.md` に書き込み：

```
# 概念追跡：{概念}

> 分析時間：YYYY-MM-DD HH:MM

## 初出
この概念が最初に登場したファイルと文脈。関連段落を最大 3 文引用。

## 進化タイムライン
- YYYY-MM-DD | [[ファイル名]] | そのファイルでの使われ方/立場の1文要約

## 現在の状態
この概念がどのトピックに接続しているか、矛盾する使われ方、発展中の draft があるか。

## 空白地帯
この概念の中で vault でまだ探求されていない側面。
```

**追記モード** — 既存ファイルの末尾に追加：

```
---

## 追跡更新 [HH:MM]

### 進化タイムライン（新規）
- [前回の分析以降の新しい言及]

### 状態の変化
- [何が変わったか。変化がなければ「顕著な変化なし」と記入]

### 空白地帯（更新）
- [未探求の側面を再評価]
```

#### Bridge モード — 2 テーマ間の隠れたつながり

スキャン：vault 全体。

ロジック：
1. `{analysis}/bridge/` で今日の同テーマペアのファイルを確認（順序不問）
2. マッチあり → 追記；なし → 新規作成

**新規作成モード** — `{analysis}/bridge/YYYY-MM-DD-{テーマA}-と-{テーマB}.md` に書き込み：

```
# 橋接分析：{テーマA} と {テーマB}

> 分析時間：YYYY-MM-DD HH:MM

## 直接の交点
両テーマに言及しているファイル。各件：[[ファイル名]]・2 つのテーマがそのファイルでどう関連しているか。
なければ「直接の交点なし」と記入。

## 橋渡しノート
片方のテーマのみ言及しているが、リンクを形成しうるファイル。なぜ橋渡しになりうるか説明。
上限 5 件、各件 [[wiki link]] で引用。

## 潜在的なつながり仮説
深層のつながりに関する 1〜3 件の仮説。各件：強 / 中 / 弱の信頼度と根拠。
```

**追記モード：**

```
---

## 追跡更新 [HH:MM]

### 新しい交点
- [前回の分析以降の新しい交点または橋渡しノート]

### 仮説の検証
- [以前の仮説を支持または反証する新しい証拠]
```

#### Ghost モード — あなたの語り口で回答

スキャン：全コンテンツフォルダ、`drafts/` と `thoughts/` サブディレクトリを除く（完成記事のみ）。

入力：`save` があれば取り除き、残りを質問として使用。

ステップ：
1. 文体の分析：文の好み・語彙の選択・論述構造・修辞的習慣
2. 質問に関連する既存のスタンスと論点を抽出
3. そのスタイルとスタンスで回答を構成

出力：

**回答** — ユーザーの語り口で。長さはユーザーの典型的な記事の段落の長さに合わせる。

**根拠** — 参照または引用したノート：[[ファイル名]]・引用した具体的な段落またはスタンス。上限 5 件。

**信頼性の注記** — どの部分が明確な vault の根拠を持ち、どの部分がスタイルから推測した延長か。

書き込み先：`{analysis}/ghost/YYYY-MM-DD-{質問の要約}.md`
同日・同質問：`## 追加回答 [HH:MM]` ブロックで追記。

#### Stress-test モード — 論述の圧力テスト

入力：`test:` プレフィックスを取り除き、`save` があれば取り除き、残りをテーマとして使用。

スキャン：テーマに言及しているすべての vault ファイル。

ステップ：
1. テーマに関するすべての論点と主張を収集
2. 各主張の論理構造と証拠を検証

出力：

**主張リスト** — vault で見つかったすべての主要な主張。各件：1 文の主張・出典 [[ファイル名]]。

**弱点分析** — 各主張について確認（該当するものだけ列挙）：
- 内部矛盾：異なるファイルが同じことについて異なる主張をしている
- 未検証の前提：証明されていない前提の上に立つ主張
- 論理のギャップ：論証の中間ステップが欠けている
- 証拠のギャップ：主張を裏付けるデータや事例が不足
各弱点は具体的な [[ファイル名]] と段落を引用。

**総合評価** — 堅牢度：堅固 / おおむね堅固だがギャップあり / 大幅な補強が必要。優先的に対処すべき 1〜3 つの弱点。

書き込み先：`{analysis}/stress-test/YYYY-MM-DD-{テーマの要約}.md`
同日・同テーマ：`## 追跡更新 [HH:MM]` ブロックで追記。

#### 共通ルール

- 20 を超えるファイルが見つかった場合は上位 20 件をリストし、総数を注記
- すべてのファイル参照は [[wiki link]] 形式
- タイムスタンプはローカル時間 HH:MM（24 時間制）
- save の有無にかかわらず、端末に完全な分析結果を出力
- 書き込み時：プレビューと完全パスを表示、確認後に実行、実行後に実際のパスを表示
- フォルダが存在しない場合は作成して完全パスを表示

## アクションプランニング

### `/hirameki:harvest [save]`

**目的：** 既存コンテンツから実行可能なアイデアを収穫。
**入力：** オプションの `save` でファイルに書き込み。
**必要なフォルダ：** analysis・daily-notes・inbox + 全コンテンツフォルダ。

スキャン：全コンテンツフォルダ・daily-notes（直近 30 日）・全 inbox。

出力 — 7 カテゴリ、各最大 5 件：

**書ける記事** — 今すぐ記事として発展させるだけの素材があるテーマ。各件：提案タイトル・素材ソース（[[ファイル名]] リスト）・執筆前に何が必要か。

**作れるツールまたはプロジェクト** — ノートで言及されたツールのニーズ・ワークフローの痛点・具体的なプロダクトアイデア。各件：説明・ソース・推定複雑度（小 / 中 / 大）。

**研究すべきテーマ** — 言及されたが深掘りされていない外部の概念・理論・技術。各件：テーマ・vault での文脈・なぜ研究する価値があるか。

**連絡すべき人またはコミュニティ** — 言及されていて現在の方向性に関連する人物・組織・コミュニティ。各件：名前・vault での文脈・連絡する理由。なければ「なし」と記入。

**別のメディアで表現したいアイデア** — 別のフォーマット（動画・ビジュアル・講演・ニュースレター・podcast など）の方が向いているコンテンツ。各件：アイデアの要約・推奨メディア・なぜそのフォーマットがテキストより合うか。

**まだ取引していない価値** — まだ収益化されていない専門知識やスキル。各件：スキルまたは知識の説明・可能な収益化の形（コース・コンサルティング・プロダクト・ライセンス）・vault の根拠。なければ「なし」と記入。

**卒業できるアイデア** — 独立したノートになれるだけの密度を持つ半成形アイデア。各件：出典場所（[[ファイル名]] + 段落の説明）・核心的な主張（1 文）・推奨する送り先コンテンツフォルダ・関連する既存ノート。

卒業の基準：明確な核心的主張がある（質問や思いつきだけではない）・vault の少なくとも 1 つの既存テーマと関連している・十分なコンテンツ密度がある（1 文以上）。

### 卒業フェーズ（save の有無にかかわらず常に 2 フェーズ）

候補リストを表示後に停止し、ユーザーがどれを実行するか選択するまで待つ。

各選択されたアイデアについて：
1. 作成する完全パスを表示、再度確認を待つ
2. 選択されたコンテンツフォルダに新しい markdown ファイルを作成：
   - タイトル
   - 核心的な主張
   - 出典の文脈（どの daily note または inbox アイテムから）
   - 関連する vault ノート（[[wiki link]] 形式）
   - 発展させる方向性
3. 実行後に実際の作成パスを表示

### 書き込みロジック（$ARGUMENTS に save が含まれる場合のみ — harvest サマリーのみ、卒業は除く）

1. `{analysis}/harvest/` で今日のファイルを確認
2. あり → `## 追跡更新 [HH:MM]` ブロックで追記
3. なし → `{analysis}/harvest/YYYY-MM-DD.md` を作成

ファイル構造：

```
# アイデアの収穫

> 分析時間：YYYY-MM-DD HH:MM
> スキャン範囲：[実際にスキャンしたディレクトリ]

## 書ける記事
[リスト]

## 作れるツールまたはプロジェクト
[リスト]

## 研究すべきテーマ
[リスト]

## 連絡すべき人またはコミュニティ
[リスト]

## 別のメディアで表現したいアイデア
[リスト]

## まだ取引していない価値
[リスト]

## 卒業できるアイデア
[リスト]
```

## メンテナンス

### `/hirameki:tidy [fix]`

**目的：** frontmatter プロパティの監査とクリーンアップ。
**入力：** オプション — 確認後に自動修正する `fix`。
**必要なフォルダ：** 全コンテンツフォルダ + inbox + daily-notes。

スキャン：
- 全コンテンツフォルダ（再帰的）
- inbox — 全ファイル
- daily-notes — 直近 30 日

**チェック項目：**

1. **欠落フィールド** — frontmatter のないファイル、`tags` または `status` が欠けているもの
2. **一貫性** — 無効な `status`/`source` 値（source の許可値：self・claude-code・agent・external）、タグの大小文字の不一致、アンダースコア vs ハイフンの不一致
3. **冗長性** — 1 回しか使われていないタグ（タイポの可能性）、6 個以上のタグを持つファイル、`topic` がタグと重複
4. **タグの収束** — 頻度統計・意味的に類似したタグのマージ候補・上位 10 個のコアタグ・孤立タグ

**出力：** 健全性レポート（問題のないファイル数 / 総ファイル数）。

```
# Frontmatter 監査レポート

> チェック時間：YYYY-MM-DD HH:MM
> スキャン範囲：[スキャンしたディレクトリ]
> スキャンファイル数：N

## 欠落フィールド（N）
- [[ファイル]] — frontmatter なし / tags なし / status なし

## 一貫性の問題（N）
- [[ファイル]] — status「xxx」は許可値に含まれない

## 冗長性の問題（N）
- [[ファイル]] — タグ 8 個、削減を推奨

## タグ概要
### コアタグ（上位 10）
| タグ | 件数 |

### マージ候補
- `ai_alignment` → `ai-alignment`

### 孤立タグ（1 回のみ使用）
| タグ | 使用場所 |

## サマリー
- 要修正：N ファイル
- 健全性：N%
```

**fix モード**（入力に `fix` が含まれる場合）：

自動修正可能（完全な変更リストを表示して確認後）：
- 欠落している frontmatter スケルトンを追加
- タグの大小文字を統一（多数決）

## 作業推論ログ

### `/hirameki:journal {説明}`

**目的：** 作業ログと思考記録。
**入力：** 必須 — トピックの説明。
**必要なフォルダ：** logs + 全コンテンツフォルダ（関連ノートの検索に使用）。

スキャン：
- 現在のセッションの操作
- トピックに関連する vault ノート
- logs にある今日の既存 journal ファイル

ロジック：
1. `{logs}/` で今日の `YYYY-MM-DD-*.md` ファイルをスキャン
2. ファイル名とタイトルを入力のトピックキーワードと比較
3. 明確なマッチ → 追記モード
4. 曖昧なマッチ → 候補をリスト、ユーザーに確認
5. マッチなし → 新規作成モード

**新規作成モード** — `{logs}/YYYY-MM-DD-{トピックの要約}.md` に書き込み：

```
# {トピック}

> 作成時間：YYYY-MM-DD HH:MM

## 背景
なぜこれが起きたか。vault に関連する既存ノートがあれば [[wiki link]] で引用。

## 何をしたか
具体的な操作と意思決定、因果の流れ。

## なぜこのアプローチか
主要な意思決定の根拠。トレードオフ：何を選び、何を捨てたか。

## インスピレーションのつながり
他のアイデアとの接続、トピック横断のリンク。関連ノートに [[wiki link]]。なければ「なし」と記入。

## 改善の可能性
未探求の方向性・代替案・潜在的リスク。なければ「なし」と記入。

## 未完了とフォローアップ
フォローアップが必要な項目。すべて完了なら「保留項目なし」と記入。
```

**追記モード** — 末尾に追加：

```
---

## 追記記録 [HH:MM]

### 何をしたか
[新しい操作と意思決定]

### なぜこのアプローチか
[新しい意思決定の根拠]

### インスピレーションのつながり
[新しいつながり。なければ「なし」と記入]

### 改善の可能性
[新しい方向性。なければ「なし」と記入]
```

「未完了とフォローアップ」セクションも確認 — 完了した項目には完了タイムスタンプを付ける。

## 書き込み動作一覧

| コマンド | 書き込み先 | トリガー | 同日重複 |
|----------|-----------|---------|---------|
| `/hirameki:wrap` | daily-notes | 常時 | 新しい Wrap ブロックを追記 |
| `/hirameki:journal` | logs | 常時 | 同テーマは追記、別テーマは新規作成 |
| `/hirameki:explore` | analysis/arc, bridge, ghost, stress-test | `save` のみ | 同概念/質問は追記 |
| `/hirameki:harvest` | analysis/harvest | `save` のみ | 追記更新 |
| `/hirameki:harvest`（卒業） | コンテンツフォルダ | 確認後 | 毎回独立 |
| `/hirameki:tidy` | analysis/tidy | 常時 | 追記更新 |

## 共通ルール

- すべてのタイムスタンプはローカル時間 HH:MM（24 時間制）
- すべてのファイル参照は [[wiki link]] 形式
- すべての書き込みコマンドは実行前にプレビューと完全パスを表示、確認後に実行
- すべての書き込みコマンドは実行後に実際の書き込みパスを表示
- 出力言語は `~/.claude/CLAUDE.md` の `## Vault Structure` から読み込む
