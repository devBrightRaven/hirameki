# Hirameki 指令 — 完整規格

所有 Hirameki 指令的完整 prompt 規格。這是人類閱讀用的參考文件 — 實際的指令 prompt 在 `commands/` 資料夾中。

---

## 共用：`__init`

所有指令開始時讀取 `~/.claude/CLAUDE.md` 的 `## Vault Structure` 段落。

### Vault 偵測

1. 檢查當前工作目錄是否有 `.obsidian/` — 如果有，這裡就是 vault
2. 如果沒有，讀取 `~/.claude/CLAUDE.md` 中的 `## Vault` 段落的 `path` 設定
3. 如果都沒有，問使用者 vault 路徑，確認包含 `.obsidian/`，然後存入 `~/.claude/CLAUDE.md`

### 語言偵測

讀取 `~/.claude/CLAUDE.md` 中 `## Vault Structure` 下的 `language` 設定。所有輸出使用這個語言。

### 資料夾偵測

每個用途按順序匹配第一個存在的候選資料夾：

| 用途 | 候選名稱 |
|------|----------|
| daily-notes | `Daily/`, `_daily/`, `daily/`, `Journal/`, `journal/` |
| inbox | `Inbox/`, `_inbox/`, `inbox/`, `_Capture/`, `Capture/` |
| analysis | `_hirameki_analysis/`, `_agent_analysis/`, `_claude_code_analysis/`, `Analysis/`, `_analysis/`, `analysis/` |
| logs | `_hirameki_logs/`, `_agent_logs/`, `_claude_code_logs/`, `Logs/`, `_logs/`, `logs/` |
| templates | `Templates/`, `_templates/`, `templates/` |

找不到時：問使用者、建立資料夾、印出完整路徑、存入 `~/.claude/CLAUDE.md` 的 `## Vault Structure`。

之後再執行時，直接讀取快取設定，不再偵測。

### 內容資料夾

「內容資料夾」= vault 根目錄下的所有第一層資料夾，排除：
- 以 `.` 開頭的資料夾（`.obsidian/`、`.claude/`、`.git/` 等）
- `_hirameki_cmds/`
- `## Vault Structure` 中列出的系統資料夾

若無內容資料夾，改掃描 vault 根目錄下的所有 `.md` 檔案。

## Session 開始

### `/hirameki:catchup [天數]`

**用途：** 進度銜接。
**輸入：** 可選天數（預設 1）。
**需要的資料夾：** daily-notes、inbox。

讀取：
- 最近的 daily notes（預設昨天，或指定天數）
- 所有 inbox 檔案

邏輯：找每份 daily note 的最後一個 Wrap 區塊。提取「進行中」和「下一步」。

輸出：

**近期進度** — 最後 Wrap 的進行中和下一步。如果沒有 daily note 或沒有 Wrap 區塊，標註「找不到進度紀錄」。多天時：按時間倒序列出每天的最後 Wrap。

**Inbox 待處理** — Inbox 檔案：檔名、建立日期、一句話摘要。上限 10 個。如果為空：「Inbox 已清空」。

**建議焦點** — 今天值得推進的 1–3 件事。每件附推薦理由（為什麼今天，而不是之後）。最多 3 件。

## Session 結束

### `/hirameki:wrap [描述]`

**用途：** 進度快照，寫入 daily note。
**輸入：** 可選重點描述。
**需要的資料夾：** daily-notes、templates。

掃描：
- 本次 session 的檔案操作（建立、修改、刪除）
- 今天的 daily note

寫入目標：`{daily-notes}/YYYY-MM-DD.md`
- 檔案不存在且有 `templates/daily.md` → 使用模板
- 檔案不存在且無模板 → 建立 `# YYYY-MM-DD` 標題
- 檔案存在 → 在末尾追加新的 Wrap 區塊，前面加水平分隔線

每個 Wrap 的格式：

```
---

## Wrap [HH:MM]

### 已完成
- [本次期間完成的具體項目，每條一句話]

### 進行中
- [已開始但未完成的項目。如果沒有，標註「無」]

### 下一步
- [接下來要做什麼。如果未決定，標註「待定」]
```

規則：
- 時間戳使用本地時間 HH:MM（24 小時制）
- 只追加，不修改之前的 Wrap 區塊
- 有輸入描述時，用它來組織 Wrap 內容
- 寫入前顯示預覽和完整路徑，等確認後再執行
- 寫入後印出實際寫入的完整路徑

## 查看 Vault 狀態

### `/hirameki:status [week|patterns]`

**用途：** Vault 現況（三種模式）。
**輸入：** 無 / `week` / `patterns`
**需要的資料夾：** daily-notes + 所有內容資料夾（patterns 模式還需要 inbox）。

#### 無參數模式 — 即時概覽

掃描：
- 所有內容資料夾（深度 2 層）
- daily-notes 最近 7 天

輸出：

**內容主題** — 每個內容資料夾及其子目錄：名稱、筆記總數、草稿數、最近修改日期。狀態：活躍（7 天內有修改）/ 休眠。

**近期活動** — 最近 7 天修改的檔案，按時間倒序。每筆：檔名、所屬資料夾、修改時間、變更性質（新建 / 修改 / 大幅改寫）。上限 15 筆。

**Vault 概覽** — 總檔案數、內容資料夾數量、最近 7 天活躍的資料夾。

空的區塊標註「無」，不跳過。

#### `status week` — 週回顧與落差分析

讀取：
- 所有內容資料夾最近 7 天的修改紀錄
- 最近 7 天的 daily notes

輸出：

**本週進度** — 每個有活動的內容資料夾：名稱、推進了什麼、下一步。

**近期動態** — 上週新增或修改的筆記，哪些接近完成。

**落差分析** — 比對 daily notes 中的優先事項與實際檔案變更：
- 說了重要但沒動手的（聲稱優先但無對應檔案變更）
- 沒提到但花了時間的（有檔案變更但 daily notes 未提及）

如果 daily notes 不足 3 天，標註「紀錄不足，落差分析可能不準確」。

#### `status patterns` — 潛流與聚攏分析

掃描：所有內容資料夾、daily-notes（最近 30 天）、所有 inbox。

**潛流主題** — 繁複出現但沒有獨立文章的主題。標準：出現在 3 個以上不同檔案中，且沒有獨立文章或 draft。每個：
- 主題名稱
- 出現次數和涉及檔案數（列出最多 5 個，[[wiki link]] 引用）
- 判斷：值得展開嗎？為什麼？
上限 10 個，按出現頻率倒序。

**聚攏中的想法群** — 3 篇以上涉及相似概念但沒有共同上層分類、且撰寫時間分布在不同日期的筆記群。每個：
- 建議群組名稱
- 涉及筆記（[[wiki link]]，上限 5 個）
- 共同主題摘要（2–3 句）
- 成熟度：高 / 中 / 低
- 建議發展方向：文章 / 專案 / 概念框架 / 繼續累積
上限 5 個，按成熟度倒序。

## 深入一個特定想法

### `/hirameki:explore {輸入} [save]`

**用途：** 概念挖掘，根據輸入形式自動選擇模式。
**輸入：** 必填。形式決定模式。末尾加 `save` 可寫入檔案。
**需要的資料夾：** analysis + 所有內容資料夾。

#### 模式偵測

| 輸入格式 | 模式 |
|----------|------|
| 單一概念或關鍵詞 | **Arc** |
| 兩個主題，以「與」、「and」或「と」分隔 | **Bridge** |
| 以 `?` 或 `？` 結尾 | **Ghost** |
| 以 `test:` 開頭 | **Stress-test** |

#### Arc 模式 — 概念演化追蹤

掃描：整個 vault，優先搜尋 daily-notes 和內容資料夾。

邏輯：
1. 檢查 `{analysis}/arc/` 中今天是否已有同概念的檔案
2. 有 → 追加模式
3. 疑似相關但不確定 → 列出候選，詢問使用者
4. 沒有 → 建立模式

**建立模式** — 寫入 `{analysis}/arc/YYYY-MM-DD-{概念}.md`：

```
# 概念追蹤：{概念}

> 分析時間：YYYY-MM-DD HH:MM

## 首次出現
最早出現這個概念的檔案及脈絡。引用相關段落（不超過 3 句）。

## 演化時間軸
- YYYY-MM-DD | [[檔名]] | 該檔案中這個概念的用法或立場摘要（一句話）

## 目前狀態
這個概念連結到哪些主題、是否有矛盾的用法、是否有 draft 正在發展它。

## 空白地帶
這個概念有哪些面向在 vault 中尚未被探討。
```

**追加模式** — 在既有檔案末尾加上：

```
---

## 追蹤更新 [HH:MM]

### 演化時間軸（新增）
- [自上次分析以來新出現的提及]

### 狀態變化
- [與上次分析相比有什麼變化。如果沒有，標註「無顯著變化」]

### 空白地帶（更新）
- [重新評估尚未探討的面向]
```

#### Bridge 模式 — 兩主題間的隱藏連結

掃描：整個 vault。

邏輯：
1. 檢查 `{analysis}/bridge/` 中今天是否已有同組主題的檔案（不分順序）
2. 有 → 追加；沒有 → 建立

**建立模式** — 寫入 `{analysis}/bridge/YYYY-MM-DD-{主題A}-與-{主題B}.md`：

```
# 橋接分析：{主題A} 與 {主題B}

> 分析時間：YYYY-MM-DD HH:MM

## 直接交集
同時提及兩個主題的檔案。每筆：[[檔名]]、兩個主題在該檔案中如何被關聯。
如果沒有，標註「無直接交集」。

## 橋樑筆記
只提及其中一個主題、但內容可能構成連結的檔案。說明為什麼它可能是橋樑。
上限 5 筆，每筆用 [[wiki link]] 引用。

## 潛在連結假設
1–3 個兩個主題可能的深層連結假設。每個標註信心程度（強 / 中 / 弱）和依據。
```

**追加模式：**

```
---

## 追蹤更新 [HH:MM]

### 新發現的交集
- [自上次分析以來新出現的交集或橋樑筆記]

### 假設驗證
- [之前的假設是否有新的支持或反駁證據]
```

#### Ghost 模式 — 用你的語氣回答

掃描：所有內容資料夾，排除 `drafts/` 和 `thoughts/` 子目錄（只看已完成文章）。

輸入：如果有 `save` 先移除，剩餘部分作為問題。

步驟：
1. 分析寫作風格：句式偏好、用詞習慣、論述結構、常用修辭
2. 提取與問題相關的既有立場和論點
3. 用這個風格和立場組成回答

輸出：

**回答** — 用使用者的語氣回答。長度與使用者典型文章段落長度一致。

**依據** — 引用或參考的筆記：[[檔名]]、具體段落或立場。上限 5 筆。

**信心標註** — 哪些部分有明確的 vault 依據，哪些是根據風格推測的延伸。

寫入：`{analysis}/ghost/YYYY-MM-DD-{問題摘要}.md`
同天同問題：追加 `## 追加回答 [HH:MM]` 區塊。

#### Stress-test 模式 — 論述壓力測試

輸入：移除開頭的 `test:`，如果有 `save` 也移除，剩餘部分作為主題。

掃描：vault 中所有提及該主題的檔案。

步驟：
1. 收集所有相關檔案中關於該主題的論點和主張
2. 逐一檢驗每個主張的邏輯結構和證據基礎

輸出：

**主張清單** — 在 vault 中找到的所有主要主張。每筆：主張內容（一句話）、來源 [[檔名]]。

**弱點分析** — 對每個主張，檢查以下項目（適用的才列）：
- 內部矛盾：vault 中不同檔案對同一件事說法不一致
- 未驗證假設：主張建立在未被證明的前提上
- 邏輯跳躍：論證中缺少中間步驟
- 證據缺口：主張缺乏支撐的資料或案例
每個弱點引用具體的 [[檔名]] 和段落。

**整體評估** — 穩固程度：穩固 / 大致穩固但有缺口 / 需要重大補強。最需要優先處理的 1–3 個弱點。

寫入：`{analysis}/stress-test/YYYY-MM-DD-{主題摘要}.md`
同天同主題：追加 `## 追蹤更新 [HH:MM]` 區塊。

#### 共用規則

- 如果搜尋結果超過 20 個檔案，只列出最重要的 20 個並標註總數
- 所有檔案引用使用 [[wiki link]] 格式
- 時間戳使用本地時間 HH:MM（24 小時制）
- 無論是否 save，都在終端輸出完整分析結果
- 寫入時：顯示預覽和完整路徑，等確認後再執行，完成後印出實際路徑
- 資料夾不存在時先建立並印出完整路徑

## 行動規劃

### `/hirameki:harvest [save]`

**用途：** 從既有內容收割可行動想法。
**輸入：** 可選 `save` 寫入檔案。
**需要的資料夾：** analysis、daily-notes、inbox + 所有內容資料夾。

掃描：所有內容資料夾、daily-notes（最近 30 天）、所有 inbox。

輸出 — 七個類別，各最多 5 個：

**可以寫的文章** — 已有足夠素材可以發展的主題。每筆：建議標題、素材來源（[[檔名]] 列表）、缺什麼才能動筆。

**可以做的工具或專案** — 筆記中提到的工具需求、流程痛點、或明確的產品想法。每筆：描述、來源、預估複雜度（小 / 中 / 大）。

**值得研究的主題** — 被提及但未深入探討的外部概念、理論、或技術。每筆：主題、vault 中提及的脈絡、為什麼值得研究。

**可以聯繫的人或社群** — 被提及且與目前方向相關的人物、組織、或社群。每筆：名稱、vault 中提及的脈絡、聯繫的理由。如果沒有，標註「無」。

**適合換個媒介的想法** — 更適合以不同形式呈現的內容（影片、視覺圖表、演講、電子報、podcast 等）。每筆：想法摘要、建議媒介、為什麼這個形式比文字更合適。

**尚未變現的價值** — 你有但還沒有轉換成收益的專業知識或能力。每筆：技能或知識描述、可能的變現形式（課程、顧問、產品、授權）、vault 中的依據。如果沒有，標註「無」。

**可以畢業的想法** — 半成形但已有足夠密度可以獨立成筆記的想法。每筆：來源位置（[[檔名]] + 段落描述）、核心主張（一句話）、建議歸屬的內容資料夾、與哪些既有筆記相關。

畢業標準：有明確的核心主張（不只是隨想）、與 vault 中至少一個既有主題相關、有足夠的內容密度（不只是一句話）。

### 畢業流程（無論是否加 save，都是兩階段）

列出候選後暫停，等使用者確認哪些要執行畢業。

對每個選中的想法：
1. 顯示即將建立的完整路徑，等再次確認
2. 在對應的內容資料夾下建立新的 markdown 檔案，包含：
   - 標題
   - 核心主張
   - 出處脈絡（從哪個 daily note 或 inbox 項目發展來的）
   - 與 vault 其他筆記的關聯（[[wiki link]] 格式）
   - 待展開的方向
3. 寫入後印出實際建立的完整路徑

### 寫入邏輯（僅當 $ARGUMENTS 包含 save 時，只涵蓋主要收割摘要，不含畢業）

1. 檢查 `{analysis}/harvest/` 中今天是否已有檔案
2. 有 → 在既有檔案末尾追加 `## 追蹤更新 [HH:MM]` 區塊
3. 沒有 → 建立 `{analysis}/harvest/YYYY-MM-DD.md`

檔案結構：

```
# 想法收割

> 分析時間：YYYY-MM-DD HH:MM
> 掃描範圍：[列出實際掃描的目錄]

## 可以寫的文章
[列表]

## 可以做的工具或專案
[列表]

## 值得研究的主題
[列表]

## 可以聯繫的人或社群
[列表]

## 適合換個媒介的想法
[列表]

## 尚未變現的價值
[列表]

## 可以畢業的想法
[列表]
```

## 維護

### `/hirameki:tidy [tags|fix|full]`

**用途：** Frontmatter 屬性健檢與清理。
**輸入：** 可選模式參數，見下表。
**需要的資料夾：** 所有內容資料夾 + inbox + daily-notes。

**模式：**

| 呼叫方式 | 執行內容 |
|----------|---------|
| `tidy`（無參數） | 缺漏欄位 + 一致性檢查 |
| `tidy tags` | 僅 tag 收斂分析 |
| `tidy fix` | 缺漏欄位 + 一致性 + 自動修正 |
| `tidy full` | 全部區塊 |

只執行對應模式的區塊，報告中不出現未執行的區塊。

掃描：
- 所有內容資料夾（遞迴）
- inbox — 全部檔案
- daily-notes — 最近 30 天

**檢查區塊：**

**缺漏欄位**（tidy / fix / full）— 無 frontmatter 的檔案、缺少 `tags` 或 `status`、空 tag 陣列

**一致性**（tidy / fix / full）— 無效的 `status`/`source` 值（source 允許：self、claude-code、agent、external）、tag 大小寫不一致、底線 vs 連字號不一致、`topic` 與 tag 重複

**冗餘**（full 限定）— 超過 6 個 tag 的檔案、`created` 格式不一致

**Tag 收斂**（tags / full）— 前 10 個核心 tag、語意相似 tag 的合併候選、孤立 tag（只出現 1 次）

**輸出：** 健康度報告（無問題檔案數 / 總檔案數）。只顯示已執行的區塊。

```
# Frontmatter 健檢報告

> 模式：[tidy / tags / fix / full]
> 檢查時間：YYYY-MM-DD HH:MM
> 掃描範圍：[掃描的目錄]
> 掃描檔案數：N

## 缺漏欄位（N）          ← tidy / fix / full
...

## 一致性問題（N）         ← tidy / fix / full
...

## 冗餘問題（N）           ← full 限定
...

## Tag 概覽               ← tags / full
### 核心 Tag（前 10）
| Tag | 數量 |

### 合併候選
- `ai_alignment` → `ai-alignment`（需更新 N 個檔案）

### 孤立 Tag（只出現 1 次）
| Tag | 出現在 |

## 摘要
- 執行區塊：[列出]
- 需要修正：N 個檔案
- 健康度：N%
```

**Fix 模式**（輸入包含 `fix` 時）：

可以自動修正（顯示完整變更清單並確認後）：
- 新增缺漏的 frontmatter 框架（空 tags + `status: draft`）
- 統一 tag 大小寫（多數決）
- 統一 tag 中的底線為連字號

需要逐一確認的：
- 合併語意相似的 tag
- 精簡超過 6 個 tag 的檔案
- 刪除孤立 tag

修正後重新計算健康度分數，輸出差異摘要。

**寫入行為：** 總是將報告寫入 `{analysis}/tidy/YYYY-MM-DD.md`。同天追加更新，包含變更摘要和健康度分數變化。

每次最多回報 50 個問題。若超過，顯示總數並建議分批處理。

## 工作推理紀錄

### `/hirameki:journal {描述}`

**用途：** 工作紀錄與思考文章。
**輸入：** 必填 — 主題描述。
**需要的資料夾：** logs + 所有內容資料夾（用於搜尋相關筆記）。

掃描：
- 本次 session 的操作
- vault 中與主題相關的筆記
- 今天在 logs 中已有的 journal 檔案

邏輯：
1. 掃描 `{logs}/` 中今天的 `YYYY-MM-DD-*.md` 檔案
2. 比對檔名和標題與輸入主題的關鍵詞
3. 明確匹配 → 追加模式
4. 模糊匹配 → 列出候選，問使用者
5. 沒有匹配 → 建立模式

**建立模式** — 寫入 `{logs}/YYYY-MM-DD-HHMM-{主題摘要}.md`（HHMM 為建立時的本地時間；slug 語言依 language 設定：繁體中文用中文、日本語用日文、English 用英文；空格以連字號取代）：

```
# {主題}

> 建立時間：YYYY-MM-DD HH:MM

## 背景
為什麼發生這件事。如果 vault 有相關既有筆記，用 [[wiki link]] 引用。

## 做了什麼
具體的操作和決策，附因果敘述。

## 為什麼這樣做
關鍵決策的理由。取捨：選擇了什麼、放棄了什麼。

## 靈感連結
與其他想法的連結、跨主題的關聯。[[wiki link]] 引用相關筆記。如果沒有，標註「無」。

## 可能的改進
未探索的方向、替代方案、潛在風險。如果沒有，標註「無」。

## 未完成與後續
需要跟進的項目。如果全部完成，標註「無待辦項目」。
```

**追加模式** — 在末尾加上：

```
---

## 追加紀錄 [HH:MM]

### 做了什麼
[新的操作和決策]

### 為什麼這樣做
[新的決策理由]

### 靈感連結
[新的連結。如果沒有，標註「無」]

### 可能的改進
[新的方向。如果沒有，標註「無」]
```

同時檢查「未完成與後續」段落 — 如果項目已完成，標記完成時間戳。

## 決策支援

### `/hirameki:decide {主題}`

**用途：** 決策前掃描。承諾之前，掃描 vault 相關脈絡，呈現決策結構。
**輸入：** 必填 — 正在考慮的決定。
**不寫入檔案。** 想保存請接著執行 `/hirameki:journal`。

掃描：所有內容資料夾 + 最近 14 天 daily notes 和 journal。

**三層輸出：**

**現況** — vault 找到的相關脈絡。包含可逆性判斷：
- **雙向門**：做錯可以回頭，偏向直接行動
- **單向門**：難以回頭，值得停留更久

**卡點** — 用反轉法找張力：什麼條件會讓這個決定確定搞砸？從那裡反推卡在哪。

**關鍵問** — 一個問題。如果知道這件事，決定就清楚了。不是建議，不是選項清單。

```
# 決策掃描：{主題}

> 掃描時間：YYYY-MM-DD HH:MM
> 相關脈絡：[[筆記1]], [[筆記2]]
> 可逆性：雙向門 / 單向門

## 現況
[vault 脈絡，有引用來源]

## 卡點
[張力所在，反轉法找出的失敗條件]

## 關鍵問
[一個問題]
```

規則：vault 裡幾乎沒有相關脈絡時，直接說明，不要填充。不寫「建議你...」。關鍵問只能是一個。
